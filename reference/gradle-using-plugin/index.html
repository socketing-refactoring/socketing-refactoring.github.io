플러그인 사용하기

Gradle의 많은 기능을 플러그인을 통해 제공된다. Gradle 배포판에 포함된 core plugin, third-party plugin, 빌드 내에서 정의된 script plubin이 있다.

플러그인은 새로운 태스크, 도메인 객체, 관습 (예: 자바 소스를 src/main/java에 위치시키기)를 도입하고, core 또는 그 이외의 플러그인 객체를 확장한다.

Gradle에 있는 플러그인은 공통 빌드 태스크를 자동화하는 데 필수적이다. 외부 도구 또는 서비스와 연동하거나 프로젝트의 특정한 요구를 충족하기 위해 빌드 프로세스를 커스텀한다. 플러그인들은 또한 빌드 로직을 만드는 주요한 메커니즘이 된다.

플러그인의 이점

빌드 스크립트에서 여러 태스크와 중복되는 설정을 작성하는 일은 복잡하다. 플러그인은 빌드 스크립트에 로직을 더하는 것 이외에도 여러 이점을 제공한다.

  • 재사용성을 증진시킴: 여러 프로젝트에서 중복되고 비슷한 로직을 사용하지 않아도 됨
  • 모듈화를 향상시킴: 더 모듈화되고 정돈된 빌드 스크립트를 가능하게 함
  • 로직 캡슐화: imperative 로직을 분리하여 declarative 빌드 스크립트를 가능하게 함

플러그인 종류

  1. BuildSrc와 컨벤션 플러그인 precompiled 플러그인과 binary 플러그인의 혼합이다. 여러 프로젝트에서 복잡한 로직을 재사용할 수 있게 하고 빌드 로직을 더 조직화되게 만들 수 있게 해준다.
// buildSrc/src/main/kotlin/shared-build-conventions.gradle.kts

plugins {
    java
}

repositories {
    mavenCentral()
}

dependencies {
    testImplementation("org.junit.jupiter:junit-jupiter:5.8.1")
    implementation("com.google.guava:guava:30.1.1-jre")
}

tasks.named<Test>("test") {
    useJUnitPlatform()
}

tasks.register<Copy>("backupTestXml") {
    from("build/test-results/test")
    into("/tmp/results/")
    exclude("binary/**")
}

// app/build.gradle.kts

plugins {
    application
    id("shared-build-conventions")
}

플러그인 사용하기

플러그인에 캡슐화된 빌드 로직을 사용하기 위하여 Gradle은 두 가지 단계를 수행해야 한다.
첫째, Gradle은 플러그인을 resolve해야 한다. 둘째, Gradle은 주로 Project를 이용하여 타켓에 플러그인을 추가해야 한다.

  1. 플러그인을 resolve한다는 것은 주어진 플러그인을 포함하고 있는 JAR의 정확한 버전을 찾아서 스크립트 classpath에 추가한다는 것을 의미한다. 플러그인이 resolve되면 플러그인의 API를 빌드 스크립트에서 사용할 수 있다.

  2. 플러그인을 추가한다는 것은 프로젝트에 플러그인의 Plugin.apply(T)을 실행한다는 것을 의미한다.

플러그인 resolve와 적용을 한 단계로 끝내고 싶다면 plugin DSL 사용이 권장된다.
